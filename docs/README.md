# OctoAcme Project Management Docs

Welcome! This README serves as the entry point to OctoAcme's project management documentation. It summarizes key workflows, roles, and practices, and links to all detailed docs in this directory.

## Overview

OctoAcme follows an iterative, customer-first delivery model built on five guiding principles: prioritizing customer value, delivering in small testable increments, maintaining clear ownership, making data-informed decisions, and fostering psychological safety. Every cross-functional project moves through a structured lifecycle—Initiation, Planning, Execution, Release, and Retrospective & Continuous Improvement—supported by lightweight but consistent artifacts such as a project one-pager, sprint backlog, risk register, and release notes.

Key personas drive each project: the **Project Manager** coordinates schedules, risks, and communications; the **Product Manager** defines outcomes and prioritizes the backlog; **Developers** implement and test features; **QA** validates acceptance criteria and quality standards; and **Stakeholders** provide inputs and approvals. Clear role definitions ensure accountability at every stage, from stakeholder alignment during initiation through post-release retrospectives.

Communication is structured around a predictable cadence—weekly PM/PdM syncs, twice-weekly delivery standups, monthly stakeholder updates, and milestone-based demos—complemented by standardized templates for weekly status reports and incident communications. Risks are tracked in a living Risk Register, assessed by impact and likelihood, and reviewed at each weekly sync to ensure mitigations stay current.

Quality assurance is embedded throughout execution: unit, integration, and end-to-end tests run in CI alongside security scanning, with manual QA for feature acceptance when needed. Releases are categorized as Patch, Minor, or Major, each requiring passing CI, completed acceptance criteria, drafted release notes, and a documented rollback plan before deployment. After every sprint or release, structured retrospectives capture what went well, what can improve, and convert learnings into tracked action items to drive continuous improvement.

## Summary

- **Principles:** Customer-first, iterative delivery, clear ownership, data-informed decisions, psychological safety.
- **Lifecycle:** Initiation → Planning → Execution → Release → Retrospective & Continuous Improvement.
- **Core Roles:** Project Manager, Product Manager, Developers, QA/Testing, Stakeholders.
- **Key Artifacts:** Project one-pager, roadmap, sprint backlog, Definition of Done, risk register, release notes, retrospective action items.
- **Cadence:** Weekly PM/PdM syncs, twice-weekly standups, monthly stakeholder updates, end-of-sprint demos.
